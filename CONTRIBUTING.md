# Contributing to Microconsortium

Thank you for your interest in contributing! This document provides
guidelines for contributing to Microconsortium projects.

## Code of Conduct

All contributors are expected to follow our [Code of Conduct](./CODE_OF_CONDUCT.md).
By participating, you agree to uphold these standards.

## How to contribute

### Reporting bugs

1. Search [existing issues](https://github.com/microconsortium) to avoid duplicates
2. Use the issue template in the relevant project repository
3. Include reproduction steps, expected vs. actual behavior, and environment details

### Suggesting features

1. Open an issue describing the feature and its motivation
2. Wait for maintainer feedback before starting work
3. We may decline features that don't fit project scope — please don't take
   it personally

### Submitting code

1. **Fork** the relevant repository
2. **Branch** from `main` with a descriptive name (e.g., `fix/parser-edge-case`)
3. **Commit** with clear messages — follow [Conventional Commits](https://www.conventionalcommits.org/)
   where possible
4. **Test** your changes — add or update tests as needed
5. **Push** to your fork
6. **Pull request** against `main` with description of changes

### Code review process

- All PRs require approval from at least one maintainer
- We aim to provide initial feedback within 7 days
- Be patient — maintainers are volunteers
- Address review comments promptly to keep momentum

## Development setup

Each project has its own setup instructions. See the project's own README.

## Licensing

By contributing, you agree that your contributions will be licensed under the
same license as the project you're contributing to. See individual project
LICENSE files.
